Construire des blocs de la plateforme .NET

C# et la plateforme .NET on été introduit en 2002 et étaient supposés offrir un modèle de programmation bien plus puissant, plus flexible et plus simple. Du point de vue du programmeur, .NET peut être vu comme un environnement d'exécution et une librairie de classes de base complète.

La couche d'exécution est appelée Common Language Runtime, ou CLR. Le rôle principal de CLR est de localiser, charger, et gérer les objets .NET en votre nom. Le CLR s'occupe aussi d'un certain nombre de détails de bas niveau, comme la gestion mémoire, l'hébergement d'applications, et les vérifications de sécurité (entre autres détails de bas-niveau).

En plus de CLR, la plateforme .NET fournit une librairie de classes de base qui est disponible à tous les langages de programmation .NET.

Un autre bloc de construction de la plateforme .NET est le Common Type System, ou CTS. La spécification CTS décrit tous les types de données possibles et toutes les constructions de programamtion supportées à l'exécution, spécifie comment ces entités peuvent interargir entres-elles.

Le rôle de CLR

  • charger le programme en mémoire
  • compiler les CIL associés en des instructions spécifiques à la plateforme
  • exécuter le code
  • gérer la mémoire
  • libérer la mémoire inutilisée (garbage collection)

La structure switch

L'instruction switch est une instruction de contrôle de flux, comme l'instruction if. Elle permet de sélectionner une séquence de code à exécuter selon la valeur d'une expression. Le mot-clé switch est utilisé pour tester la valeur d'une expression par rapport à une liste de valeurs. La liste de valeurs est présentée avec le mot-clé case. Si les valeurs correspondent, le bloc d'instruction qui suit le case est exécuté.

Le compilateur C# requière le mot-clé break à la fin de chaque section case contenant du code. Si aucun code n'est trouvé après une instruction case, le break peut être omis, et l'exécution passe à la prochaine instruction case et continue jusqu'à ce qu'il trouve le mot-clé break. Après la structure default, le mot clé break est obligatoire.

En C#, le type de l'expression sur laquelle porte le choix peut être un entier, char, string ou enum.

Conversions implicites de type

Un conversion de type implicite (cachée) est possible quand il n'y a aucun risque de perte de données durant la conversion, c'est-à-dire quand on convertit un type d'une certaine étendue vers un type d'une étandue plus grande (par exemple de int vers long). La conversion implicite est faite automatiquement par le compilateur quand on assigne une valeur avec une étandue plus petite dans une variable avec une étandue plus grande ou si l'expression possède différents types avec des étandues différentes. Dans un tel cas, la conversion est effectuée dans le type qui possède l'étendue la plus grande.

Il n'y a pas de perte de données quand on convertit des types de plus petites étendues vers des types de plus grandes étendues.

static void ImplicitConversionTest()
{
	int myInt = 5;
	Console.WriteLine(myInt);
	long myLong = myInt; // conversion implicite
	Console.WriteLine(myLong);
	Console.WriteLine(myLong + myInt); // conversion implicite
}

Mot clé const

Pile

C'est une zone spéciale de la mémoire de votre ordinateur qui stocke les variables temporaires créées par chaque fonction (incluant la fonction main()). La pile est une structure de données LIFO (Last In, First Out, donc dernier arrivé, premier sorti), qui est gérée et optimisée par le CPU assez étroitement. Chaque fois qu'une fonction déclare une nouvelle variable, elle est "poussée" sur le haut de la pile. Ensuite, chaque fois qu'une fonction se termine, toutes les variables poussées sur la pile par cette fonction sont libérées (nous dirons, supprimées). Une fois qu'une variable de la pile est libérée, cette région de la mémoire devient disponible pour d'autres variables de pile.

L'avantage d'utiliser la pile pour stocker les variables, et que cette mémoire est gérée pour nous. Il n'est pas nécessaire d'allouer la mémoire à la main, ou de la libérer une fois qu'elle n'est plus utile.

Une autre caractéristique de la pile qu'il faut garder en tête, est qu'il y a une limite (qui varie avec le système d'exploitation) sur la taille des variables que l'on peut mettre dans cette zone mémoire.

La pile est utilisée pour stocker
  • Les variables locales des méthodes
  • Les paramètres des méthodes
  • L'adresse de retour des méthodes

Principes fondamentaux sur les méthodes en C#

Une méthode est une partie basique d'un programme. Elle peut résoudre un certain problème, éventuellement prendre des paramètres et retourner un résultat.

Une méthode représente toutes les conversions de données qu'un programme peut effectuer pour résoudre une tâche particulière. Les méthodes constituent la logique du programme.

Dès lors qu'un programme est créé, c'est une bonne pratique que d'utiliser des méthodes. Cela permet de créer du code plus structuré et plus facile à lire.

Déclaration et appel d'une méthode

  • En C#, une méthode doit être déclarée à l'intérieur d'une classe.
  • Une méthode ne peut être déclarée dans le corps d'une autre méthode.
  • Les éléments nécessaires pour déclarer une méthode :
    • Le type du résultat retourné par la méthode
    • Le nom de la méthode
    • La liste des paramètres de la méthode, cela peut être une liste vide ou une suite de déclarations de paramètres.
    • Exemple: 
      class Calculator {
    public int Add2Integers(int n1, int n2) {
        return n1 + n2;
    }
}
  • Convention de nommage pour les méthodes en C#
    • Une méthode C# commence avec une lettre majuscule.
    • La règle PascalCase est appliquée. C'est à dire que chaque nouveau mot, qui constitue une partie du nom de la méthode, doit commencer par une lettre capitale.
    • Il est recommandé que le nom de la méthode soit constitué d'un verbe, ou d'un verbe et d'un nom.
  • Invocation ou applel de méthode
    • Invoquer ou appeler une méthode est en fait le processus d'exécution du code de la méthode.
    • Exemple: 
      class Program
{
    static void Main(string[] args)
    {
        int number1 = 2, number2 = 4; ;

        int sum = cal.AddIntegers(number1, number2);

        Console.ReadLine();
    }
}
  • En C#, l'ordre des méthodes dans la classe n'est pas important. Il est autorisé d'appeler une méthode avant qu'elle ne soit déclarée dans le code.

La surcharge de méthode

Quand on définit un ensemble de méthodes du même nom qui ne diffèrent que par le nombre (ou type) de paramètres, la méthode en question est dite surchargée. La signature d'une méthode est spécifiée par : les types de paramètres et l'ordre dans lequel les paramètres sont listés. Le nom des paramètres de la méthode ne sont pas significatifs pour la déclaration de la méthode. Les surcharges en C# ne sont autorisées que pour des méthodes avec des signatures différentes. Le type de retour ne fait pas partie de la signature de la méthode, et ne peut donc pas être utilisé pour différencier des surcharges.

Exemple de surcharge de méthode en C# 
class Calculator
{
    public int Add(int n1, int n2)
    {
        return n1 + n2;
    }
    public double Add(double n1, double n2) {
        return n1 + n2;
    }
}

Programmation orientée objets (POO)

La programmation orientée objets est le successeur de la programmation procédurale ou structurelle. La programmation procédurale décrit les programmes comme des groupes d'unités de code réutilisables (procédures) qui définissent les paramètres d'entrée et de sortie. Les programmes procéduraux sont constitués de procédures, qui s'invoquent les unes les autres.

Le problème avec la programmation procédurale est que la réutilisabilité du code est difficile et limitée - seules les procédures peuvent être réutilisées et il est difficile de les faire génériques et flexibles.

L'approche orientée objets se base sur le paradigme que tout programme travaille avec des données qui décrivent les entités (objets ou évènements) de la vie réelle.

Les objets décrivent les caractéristiques (champs) et le comportement (méthodes) d'objets de la vie réelle.

Les principaux avantages et buts de la POO sont de rendre les applications complexes plus rapides à développer et plus faciles à maintenir.

Principes fondamentaux de la programmation orientée objets
  • Encapsulation
    - pour cacher les détails non nécessaires dans nos classes et fournir une interface claire et simple pour travailler avec elles.
  • Héritage
    - pour permettre la réutilisabilité de fonctionalités et améliorer la lisibilité du code.
  • Polymorphisme
    - pour assigner une signification ou un usage différent à certains comportements, selon le contexte.

Définitions des classes en C#

Une classe en POO est appelée une définition (ou spécification) d'un certain type d'objets du monde réel. La classe représente un modèle, qui décrit les différents états et comportements des objets qui sont créés à partir de cette classe (modèle).

Un objet est une copie créée à partir de la définition (spécification) d'une classe donnée. Cet objet est aussi appelé une instance.

Une classe en C#

  • est un nouveau type
  • est de type référence
  • devrait ête déclarée un espace de noms (namespace)

C# autorise les espaces de noms multiples et de multiples classes dans un seul fichier source. En C#, contrairement à Java, le nom du fichier n'est pas nécessairement le même que le nom de la classe.

Membres de classes ou éléments

Les membres qu'une classe peut contenir :
  • Champs
    - Les champs sont des variables déclarées dans la portée de la classe.
  • Propriétés
    - Les propriétés sont des méthodes de classe. On y accède comme si elles étaient des champs de cette classe. Une propriété peut fournir une protection pour un champs de classe.
  • Méthodes
    - Les méthodes définissent les actions qu'une classe peut effectuer.
  • Évènements
    - Les évènements fournissent des notifications concernant les occurences, tels que des clics de bouton ou la fin d'exécution d'une méthode, à d'autres objets. Les évènements sont définis et déclenchés grâce aux delegates.

Les modificateurs d'accèes pour les membres de classes

En C#, il existe quatre modificateurs d'accès : public, private, protected et internal.

  • private
    - Les membres privés ne sont pas accessibles à partir d'autres classes (sauf la classe dans laquelle ils sont définis).
    - Les membres de classes sont privés par défaut si aucun modificateur d'accès n'est spécifié.
  • public
    - Les membres publics sont accessibles de n'importe qu'elle classe, quel que soit le projet courant (assembly), à partir de l'espace de nom courant.
  • internal
    - Le modificateur internal est utilisé pour limiter l'accès aux éléments de la classe aux fichiers du même assembly, c'est à dire le même projet dans Visual Studio. Quand on crée plusieurs projets dans Visual Studio, les classes d'un projet seront compilées dans un assembly spécifique au projet.
  • protected
    - Les membres de classes protégés sont visibles à toutes les classes héritières (décendants).
  • protected internal
    - Les membres internes protégés sont accessibles à partir de l'assembly dans lequel ils sont déclarés, ou de l'intérieur de classes dérivées dans un autre assembly.

Les modificateurs d'accès pour les types (structures et classes)

Les classes et les structures qui sont déclarées directement dans un espace de noms peuvent être public ou internal.

  • public
    - Le type est accessible par tout code dans le même assembly ou un autre assembly qui y fait référence. -
  • internal
    - Le type ou membre est accessible par tout code dans le même assembly, mais pas d'un autre assembly.
    - Les types (classes et structures) sont internal par défaut si aucun modificateur d'accès n'est spécifié.

Les classes dérivées ne peuvent avoir une accessibilité plus grande que leurs classes de base.

Les membres énumérés sont toujours public, et aucun modificateur d'accès ne peut être appliqué.

Mot clé this

Le mot clé this fait référence à l'instance courante du type.

Exemple 
class Employee
{

    public void printEmployee()

Un constructeur d'un type
  • sert à initialiser la mémoire allouée pour l'objet, là où ses champs seront stockés (ceux qui ne sont pas statiques),

  • n'a pas de type de retour,
  • est appelé automatiquement quand on utilise le mot clé new,

  • peut appeler un autre constructeur.

Destructeurs

Les destructeurs sont utilisés lors de la destruction des instances de classes.

Le destructeur d'une classe
  • Une classe ne peut avoir qu'un seul destructeur.
  • Les destructeurs ne peuvent être ni hérités, ni surchargés.
  • Les destructeurs ne peuvent pas être appelés. Ils sont invoqués automatiquement.
  • Un destructeur n'a pas de modificateur d'accès ni de paramètres.
  • Les destructeurs ne peuvent être définis dans des structures (struct). Ils ne sont utilisés que pour les classes.

Un membre statique d'un type est un champ, propriété, méthode ou autre membre, qui contient le modificateur static dans sa déclaration. Cela signifie que ce membre, déclaré comme statique, appartient à la classe plutôt qu'à une instance particulière de cette classe.

Héritage

L'héritage
  • permet de réutiliser des fonctionnalités et d'améliorer la lisibilité.
  • permet de définir des fonctionnalités communes dans une classe parent (ou classe de base) qui peuvent être utilisées, et éventuellement modifiées, dans des sous-classes (ou classes filles).
  • est décrit comme étant une relation est-un, par exemple, un tigre est-un animal.
  • En .NET, une classe ne peut hériter que d'une seule classe (héritage simple), contrairement à C++ qui permet qu'une classe hérite de plusieurs classes (héritage multiple).
  • La classe de base de toutes les classes dans le framework .NET est System.Object. Elle constitue la racine de la hierarchie de classes.

Mot clé base

Le mot clé base est utilisé pour accéder aux membres de la classe de base dans une classe dérivée:

  1. pour spécifier quel constructeur de la classe de base doit être appelé quand des instances de la classe dérivée sont créées.
  2. pour appeler une méthode de la classe de base.
spécifier le constructeur de la classe de base

Quand on hérite d'une classe, les constructeurs doivent appeler un constructeur de la classe de base, de façon à initialiser ses données membres. Si on ne fait pas cela explicitement, le compilateur va effectuer un appel au construcuteur sans paramètre de la classe de base, ":base()"

public class ExtendingClass : BaseClass
{
public ExtendingClass() { … }
}
Cela ressemble en fait à cela : 
public class ExtendingClass : BaseClass
{
public ExtendingClass() : base() { … }
}

Si la classe de base n'a pas de constructeur sans paramètre ou que ce constructeur est caché, les constructeurs doivent appeler explicitement l'un des autres constructeurs de la classe de base. L'ommission d'un tel appel entraînera une erreur de compilation.

Référence à la classe de base et à la classe dérivée

En C#, une référence à la classe de base peut référer à un objet de la classe dérivée, mais on ne pas pas utiliser une référence vers une classe de base pour accéder aux membres d'une classe dérivée, même si l'objet réel sur la tas est du type de la classe dérivée.

Mots clé virtual et override

Le mot clé virtual est utilisé pour modifier la déclaration d'une méthode, d'une propriété ou d'un évènement pour permettre qu'elle soit redéfinie dans une classe dérivée. Les méthodes virtuelles peuvent être redéfinie (avec le mot clé override) par toute classe dérivée.

Virtuelle veut dire "J'ai une implémentation par défaut mais vous pouvez me changer si vous avez besoin.

Exemple

{
    public string Name { get; set; }
    public int Age { get; set; }
    public virtual void SayHi()
    {
        Console.WriteLine("Person SayHi");
    }
}

class Chinese : Person
{
    public override void SayHi()
    {
        Console.WriteLine("Nihao");
    }
}

class TestApp {
    static void Main(string[] args)
    {
        Person person1 = new Chinese();
        Person person2 = new Person();
        person1.SayHi();
        person2.SayHi();
    }
}

Une classe dérivée peut redéfinir une membre de la classe de base seulement si le membre de la classe de base est déclaré virtual ou abstract.

À l'exécution, lors d'un appel à une méthode virtuelle, le type de l'instance sur lequel la méthode est appelée détermine la méthode qui est effectivement exécutée. Pour l'invocation d'une méthode non virtuelle, c'est le type de l'instance, à la compilation, qui est le facteur déterminant.

On peut utiliser le mot clé base pour appeler une méthode virtuelle de la classe de base de l'intérieur d'une méthode redéfinie dans la classe dérivée.

class Chinese : Person
{
    public override void SayHi()
    {
        base.SayHi();
        Console.WriteLine("Nihao");
    }
}

Les 3 conditions pour pouvoir utiliser le polymorphisme

  1. Héritage
  2. La méthode de la classe de base est virtuelle ou abstraite. La classe dérivée redéfinit la méthode virtuelle ou abstraite de la classe de base.
  3. Utiliser une référence (ou un pointeur) de la classe de base pour pointer sur un objet de la classe dérivée.

Le polymorphisme est partout dans le framework .NET.

Classes abstraites

Le modificateur abstract indique que ce qui est modifié possède une implémentation incomplète ou même ne possède pas d'implémentation du tout. Le modificateur abstract peut être utilisé pour les classes ou les méthodes.

On utilise le modificateur abstract pour une class pour signifier que la classe doit être la classe de base d'autres classes.

Les classes abstraites possèdent les fonctionnalités suivantes :

  • Une classe abstraite ne peut être instanciée.
  • Une classe abstraite peut contenir des méthodes et des propriétés abstraites.
  • Il n'est pas possible de modifier une classe abstraite avec le modificateur sealed parce que les deux modificateurs ont des sens opposés.
  • Une classe non abstraite dérivée d'une classe abstraite doit inclure les implémentations de toutes les méthodes et propriétés abstraites héritées.

    public int Id { get; set; }

Quand il est appliqué à une classe, le modificateur sealed empêche toute classe d'hériter de cette classe. Dans l'exemple suivant, la classe B hérite de la classe A, mais aucune classe ne peut hériter de la classe B.

class A {}
sealed class B : A {}

Interface

Une interface
  • est un type,
  • contient un ensemble de comportements communs (méthodes, propriétés ou évènements) qu'une classe ou une structure peuvent implémenter,
  • ne peut êter instanciée,
  • ne peut contenir de membres statiques et
  • aucune déclaration de méthodes ne possède de corps dans une interface.
Créer une interface en C#
  • On utilise le mot-clé interface.
  • Les membres d'interfaces sont automatiquement publics, et ne peuvent avoir de modificateur d'accès.
  • Pour implémenter un membre d'interface, le membre correspondant dans la classe d'implémentation doit être public et possèder le même nom et la même signature que le membre d'interface. En C#, quand on implémente une interface, on n'utilise pas le mot clé override.
interface IRunnable {
    void Run();
}

class Person : IRunnable
{
    public void Run()
    {
        Console.WriteLine("Person Running");
    }
}

Délégués

Les délégués en C# sont similaires aux pointeurs de fonctions en C ou C++. Un délégué est une type référence. Le but d'un délégué est d'encapsuler des méthodes. Quand on crée le délégué, on spécifie une signature de méthode et un type de retour; on peut alors encapsuler, dans le délégué, n'importe quelle méthode qui correspond à la signature.

Définir un type délégué en C#

Quand on veut créer un type délégué en C#, on utilise le mot clé delegate. Une fois que le délégué est défini, on peut encapsuler une méthode dans ce délégué en créant une instance du délégué et en lui passant une méthode qui correspond au type de retour et à la signature du délégué. On passe une référence sur cette méthode. Un objet de delegate peut alors être invoqué comme si il était une méthode encapsulée lui même. 

namespace DelegateTestNS {
    public delegate void DGSayHi();

    public class TestApp {
        static void Main() {
            DGSayHi dgSayHi = new DGSayHi(SayHiFr);
            if (dgSayHi != null) {
                dgSayHi();
            }
        }

        static void SayHiFr() {
            Console.WriteLine("Bonjour");
        }
    }
}

Un objet delégué peut appeler plus d'une méthode quand on l'invoque.

  • += pour ajouter une nouvelle méthode à la liste des méthodes du délégué (la liste d'invocation)
  • -= pour supprimer une méthode de la liste d'invocation
namespace DelegateTestNS {
    public delegate void DGSayHi();

    public class TestApp {

            }
            dgSayHi += new Chinese().SayHiCh;
            Canadian canadian = new Canadian();
            dgSayHi += canadian.SayHiEn;
            dgSayHi += canadian.SayHiFr;
            if (dgSayHi != null)
            {
                dgSayHi();
            }
            dgSayHi -= canadian.SayHiFr;
            if (dgSayHi != null)
            {
                dgSayHi();
            }
        }

        static void SayHiFr() {
            Console.WriteLine("Bonjour");
        }
    }

        public void SayHiCh()
        {
            Console.WriteLine("Nihao");

    public class Canadian
    {

            Console.WriteLine("Hello");

        }

Le patron de conceptions Observateur (Observer) et les évènements

Les patrons de conception

Les patrons de conception assurent que les problèmes les plus communs soient adressés par des solutions acceptées et bien connues. La force principale des patrons de conception réside dans le fait que la plupart des problèmes ont probablement été rencontrés et résolus par une autre personne ou par des équipes de développement.

Le patron de conception Observateur

L'observateur s'enregistre auprès du sujet observé, en lui exprimant son intérêt dans l'observation. Quand un état change ou qu'un évènement survient (un clic de bouton par exemple), le sujet avertit l'observateur du changement, et celui-ci peut réagir en conséquence. Quand l'observateur ne désire plus observer le sujet, il se désabonne du sujet. Ces étapes sont connues comme étant l'enregistrement de l'observateur, la notification et la supression de l'enregistrement, respectivement.

Exemple propriétaire de chien, version 1

dans cet exemple, nous allons écrire 2 classes Owner et Chien. Owner est l'observateur (observer en anglais), Dog est l'observé (observable en anglais). Un propriétaire observe un évènement alarmOn d'un chien.

Implémentation pas à pas du patron de conception Observateur avec les délégué C# :

  1. Définir un nouveau type délégué pour l'évènement. Ce type délégué sera utilisé pour envoyer des notifications à l'observateur.
  2. Dans l'observé, créer une donnée membre de ce type délégué. Ce membre sera l'évènement.
  3. Dans l'observé, créer une méthode pour invoquer la liste d'invocation du délégué dans les circonstances correctes.
  4. Dans l'observateur, créer une donnée membre du type de l'observé.
  5. Dans l'observateur, créer une méthodes (le gestionnaire d'évènement) en fonction du type de l'évènement délégué défini à l'étape 1.
  6. Dans l'observateur, enregistrer/attacher/associer le gestionnaire d'évènements à l'évènement de l'observé (défini à l'étape 2).
  7. Déclencher l'évènement au bon moment.

Structs vs. Classes

En C# et dans le framework .NET, il y a deux implémentations du concept de classe provenant de la programmation orientée objet : les classes et les structures. Les classes sont définies par le mot clé class alors que les structures sont définies avec le mot clé struct. La différence principale entre une structure et une classe est que :

  • Les classes sont des types référence.

    Quand un objet de la classe est créé, la variable à laquelle l'objet est assigné ne contient qu'une référence à cette mémoire. Quand la référence de l'objet est assignée à une nouvelle variable, la nouvelle variable réfère à l'objet original. Les changements faits à travers une variable sont reflétés dans l'autre variable parce qu'ils réfèrent tous les deux à la même donnée.

  • Les structures (structs) sont des types valeur.

    Quand une structure est créée, la variable à laquelle la structure est assignée contient les données de la structure. Quand une structure est assignée à une nouvelle variable, elle est copiée. La nouvelle variable et la variable originiale contiennent deux copies séparées des mêmes données. Les changements effectués à une copie n'affectent pas l'autre copie.

En général, les classes sont utilisées pour modéliser des comportements plus complexes, ou des données qui sont supposées être modifiées après qu'un objet de la classe soit créé. Les structs conviennent mieux pour des petites structures de données qui contiennent de données initiales, et qui ne sont pas supposées être modifiées après que la structure ait été créée.

Définir une structure

Le type struct convient pour représenter des objets légers tel que Point, Rectangle, et Couleur. Bien qu'il soit aussi intéressant de représenter un point par une classe avec des propriétés auto-implémentées, une structure peut être plus efficace dans certains cas. Par exemple, si on déclare un tableau de 1000 objets de type Point, on va allouer de la mémoire additionnelle pour référencer chaque objet; dans ce cas, une structure serait moins couteuse.

À l'intérieur d'un struct:

  • on ne peut pas définir de constructeur sans paramètre;
  • on ne peut pas initialiser un attribut d'instance dans le corps d'un struct;
  • on peut initialiser les membres d'un struct seulement en utilisant un constructeur paramétré ou en accédant aux membres individuellement après que la structure doit déclarée.
  • Tout membre privé, ou inaccessible en général, ne peut être initialisé que dans un constructeur.
  • Il n'y a pas d'héritage pour les structures comme il peut y en avoir pour les classes. Une structure ne peut hériter d'une autre structure ou d'une classe, et elle ne peut pas être la base d'une classe. Les structs, malgré tout, hérite de la classe de base Object.
  • Une structure peut implémenter des interfaces.

Mots clé try, catch et finally

L'instruction try permet de désigner des blocs de code pour les exceptions et de fournir du code pour les gérer si elles se produisent. L'instruction try consiste en trois sections :

  • Le bloc try contient les instructions qui peuvent potentiellement lever des exceptions.
  • La section des clauses catch contient une ou plusieurs clauses catch. Ce sont des blocs de code pour gérer les exceptions. On les appelle aussi les gestionnaires d'exception.
  • Le bloc finally contient du code à exécuter en toute circonstance, qu'une exception ait été levée ou non.

Pour une instruction try, le bloc try est requis. De même sont requis un ou plusieurs blocs catch associés, ou un bloc finally, ou les deux. Une instruction try sans bloc catch, ni bloc finally entraîne une erreur de compilation.

Une fois qu'une exception survient dans le bloc try, le flux d'exécution saute au premier gestionnaire d'exception associé à cette exception. 

try
{
    //code surveillé pour exceptions
}
catch (Exception ex) {
    //code de gestion d'exception
}

Une instruction try peut contenir plus d'un bloc catch. La première instruction catch qui peut gérer l'exception est exécutée; toutes les instructions catch suivantes sont ignorées, même si elles sont compatibles avec l'exception. C'est pourquoi, les blocs catch devraient toujours être ordonnés du plus spécifique (ou plus dérivé) vers le moins spécifique. 

try
{
    //code surveillé pour exceptions
}
catch (DivideByZeroException ex)
{
    //code de gestion d'exception pour l'exception DivideByZeroException
}
catch (Exception ex) {
    //code de gestion d'exception pour toute autre exception
}

Le code dans le bloc finally est exécuté même si une exception est levée. On peut alors utiliser un bloc finally pour libérer des ressources, par exemple pour fermer tout flux ou fichier qui aurait été ouvert dans le bloc try.

  • Si aucune exception ne s'est produite dans le bloc try, alors, à la fin du bloc try, toutes les clauses catch sont sautées et le bloc finally est exécuté.
  • Si une exception se produit dans le bloc try, alors la clause catch appropriée (dans la section des clauses catch) est exécutée, suivie par l'exécution du bloc finally. 
try
{
    //code surveillé pour exceptions
}
finally {
    //code devant être exécuté en toute circonstance, qu'une exception ait eu lieu ou non.
}
try
{
    //code surveillé pour exceptions
}
catch (Exception ex) {
    //code de gestion d'exception
}
finally
{
    //code devant être exécuté en toute circonstance, qu'une exception ait eu lieu ou non. 
}

Une clause catch peut être utilisée sans argument pour attraper tout type d'exception.

try
{
    //code surveillé pour exceptions
}

    //code de gestion d'exception
    Console.WriteLine("There is an error...");

Les classes d'Exception

Il y a beaucoup de différents types d'exceptions qui peuvent se produire dans un programme. En .Net, les exceptions sont des types qui, en fin de compte, dérivent tous de la classe System.Exception. Le FCL définit un nombre de classes d'exception, chacune représentant un type spécifique. Quand l'une d'elle se produit, le CLR fait ce qui suit :

  • Il crée un objet d'exception pour le type.
  • Il cherche une clause catch appropriée pour gérer l'exception.

Hierarchie d'héritage pour les exceptions

Un objet d'exception contient des propriétés avec l'information concernant l'exception qui l'a causé.

  • Message : coneitn un message d'erreur expliquant la cause de l'exception.
  • StackTrace : contient les informations décrivant l'endroit où l'exception à eu lieu.
  • InnerException : si l'exception courante a été levée par une autre exception, cette propriété contient une référence à l'exception précédente.
  • Source : Si non initialisée par une exception définie par l'application, cette propriété contient le nom de l'assembly dans lequel l'exception a pris naissance.
  • Data : Donne une collection de paires clé/valeur fournissant des informations supplémentaires définies par l'utilisateur concernant l'exception.

Introduction à ADO.NET

ADO.NET est un ensemble de librairies orienté-objets permettant d'interragir avec des sources de données. En général, la source de données est une base de données, mais peut tout aussi bien être un fichier text, Excel ou XML. Pour ce cours, nous utiliserons ADO.NET afin d'interragir avec une base de données SQL Server.

Les fournisseurs de données

ADO.NET permet d'interragir avec différents types de sources de données et différents types de base de données. Malgré tout, il n'y a pas un unique ensemble de classes qui permet d'accomplir cela de façon universelle. Puisque des sources de données différentes ont des protocoles d'accès différents, il faut une façon de communiquer avec la bonne source de données en utilisant le protocole adapté. Quelques vieilles sources de données utilisent le protocole ODBC, beaucoup de sources de données plus récentes utilisent le protocole OleDb, et il y a de plus en plus de sources de données qui permettent de communiquer directement avec elles à travers les librairies de classes .NET ADO.NET.

ADO.NET fournit un moyen relativement commun d'interragir avec les sources de données, mais cela vient dans différents ensembles de librairies selon la façon dont on veut parler avec la source de données. Ces librairies sont appelées fournisseurs de données (Data Providers) et sont généralement nommées en fonction du protocole ou du type des données source avec lesquelles elles permettent d'interragir.

La table suivante liste quelques fournisseurs de données bien connus, le préfixe d'API qu'elles utilisent et le type des sources de données avec lesquel elles vous permettent d'interagir.

SQL Data Provider Sql pour interragir avec un serveur Microsoft SQL
ODBC Data Provider Odbc les sources de données avec une interface ODBC. Généralement les anciennes bases de données.
OleDb Data Provider OleDb les sources de données avec une interface OleDb, i.e. Access ou Excel.
Oracle Data Provider Oracle pour les bases de données ORACLE.
Borland Data Provider Bdp accés générique à beaucoup de bases de données comme Interbase, SQL Server, IBM DB2 et Oracle.

Un exemple pour mieux comprendre le sens du préfixe d'API. L'un des premiers objets ADO.NET que l'on va apprendre est l'objet de connexion, qui permet d'établir une connexion avec une source de données. Si on utilisait un fournisseur de données OleDb pour se connecter à une source de données qui expose une interface OleDb, on utiliserait alors un objet de connexion nommé OleDbConnection. Puisque nous utilisons SQL Server dans ce cours, tous les objets de l'API posséderont le préfixe Sql (comme SqlConnection par exemple).

Les objets ADO.NET

ADO.NET inclue beaucoup d'objets que l'on peut utiliser pour travailler avec des données. Cette section introduit quelques uns des objets de base que nous utiliserons:

  • L'objet SqlConnection

    Pour interragir avec une base de données, on doit avoir une connexion avec elle. La connexion permet d'identifier le serveur de base de données, le nom de la base de données, le nom de l'tuilisateur, le mot de passe, et d'autres paramètres qui sont requis pour se connecter à la base de données. Un objet de connexion est utilisé par les objets de commande pour qu'ils puissent savoir sur quelle base de données executer la commande.

  • Lobjet SqlCommand

    Le processus d'interraction avec une base de données demande à ce que l'on spécifie les actions qui doivent se produire. Ceci est effectué par un objet de commande. On utilise un objet commande pour envoyer des instructions SQL à la base de données.

  • L'objet SqlDataReader

    Beaucoup d'opérations de données demandent que l'on ait un flux de données pour la lecture. Le lecteur de données permet d'obtenir les résultats d'une instuction SELECT à partir d'un objet de commande. Pour des raisons de performance, les données retournées du lecteur de données constituent un flux de données à avance rapide. C'est à dire que l'on ne peut extraire les données du flux que de manière séquentielle. Ceci est bon pour la vitesse.

  • L'objet DataSet

    Les objets de type DataSet constituent une représentation des données en mémoire. Ils contiennent de multiples objets de type DataTable, lesquels sont constitués de lignes et de colonnes, comme de simples tables d'une base de données. Le DataSet est spécifiquement fait pour aider à la gestion des données en mémoire et supporter les opérations sur les données en mode déconnecté. Le type DataSet est utilisé par tous les fournisseurs de données, c'est pourquoi il ne possède pas de préfixe d'un fournisseur de données spécifique.

  • L'objet SqlDataAdapter

    L'objet SqlDataAdapter aide à gérer les données en mode déconnecté. Le data adapter remplit un objet DataSet quand il lit les données.

Lire des données avec la classe SqlDataReader (mode connecté)

SqlDataReader est un type qui est utile pour lire des données de la façon la plus efficace possible.

À partir d'objets de type SqlDataReader, on peut lire d'une façon séquentielle. Une fois que des données ont été lues, on doit les sauvegarder car il n'est pas possible de revenir en arrière et de les relire à nouveau. Il faut rester connecté à la base de données durant la lecture des données. C'est ce que l'on appelle le mode connecté.

La conception de lecture sans retour de SqlDataReader est ce qui lui permet d'être rapide. Il n'y a pas de coût supplémentaire associé à la lecture ou à l'écriture des données. Donc, si on veut lire une seule fois un groupe de données, le plus rapidement possible, alors un SqlDataReader est le meilleur choix. De même, si la quantité de données que l'on a besoin de lire est plus grande que ce que l'on veut garder en mémoire en une seule fois, alors le comportement de flux de SqlDataReader serait un bon choix.

Pour obtenir une instance de SqlDataReader, on doit appeler ExecuteReader sur un objet SqlCommand. La méthode ExecuteReader de l'objet SqlCommand retourne une instance de SqlDataReader. Si on crée un SqlDataReader avec l'opérateur new, il ne se passera rien.

Stocker les chaînes de connexion dans un fichier de configuration de l'application

Inclure les chaînes de connexion dans le code de l'application peut conduire à des failles de sécurité et à des problèmes de maintenance. Des chaînes de connexion non encryptées dans le code source de l'application peuvent être vues en utilisant l'outil de décompilation (IL Disassembler). De plus, si la chaîne de connexion change, l'application doit être recompilée. Pour toutes ces raisons, il est recommandé de stocker les chaînes de connexion dans un fichier de configuration de l'application.

Les fichiers de configuration d'application contiennent les paramètres qui sont spécifiques à une application particulière. Par exemple, en ASP.NET, une application peut avoir un ou plusieurs fichiers web.config, et une application Windows peut avoir un fichier app.config optionnel. Les chaînes de connexion peuvent être stockées comme des pairs clé/valeur dans la section connectionStrings de l'élément de configuration d'un fichier de configuration d'applciation.

<?xml version="1.0" encoding="utf-8" ?>
<configuration>
    <startup>
        <supportedRuntime version="v4.0" sku=".NETFramework,Version=v4.6.1" />
    </startup>

    <connectionStrings>
      <add name="MyConnectionString"
           connectionString="Data Source=your_sql_server_name; Initial Catalog=MyTest; Integrated security=True"/>

WebConfigurationManager est utilisé pour travailler avec les fichiers de configuration de ASP.NET. Il est conçu pour travailler avec des fichiers de configuration sur un serveur Web, et autorise l'accès par programmation aux section du fichier de configuration comme system.web.

Architecture multicouches

Une architecture logicielle multi-niveaux est une architecture logicielle qui utilise beaucoup de couches pour distribuer les différentes responsabilités d'un produit logiciel. Les termes "tiers" ou "couche" sont souvent utilisés.

Dans une architecture multi-niveaux logique pour un système d'information suivant une conception orientée-objet, ce qui suit est le plus commun :

  • La couche interface utilisateur (UI : User Interface) : La couche de présentation fournit l'interface utilisateur de l'application (UI). Typiquement, cela induit l'utilisation de Windows Forms pour des interactions client intelligentes, et les technologies ASP.NET pour les interractions basées sur le navigateur. La couche de présentation peut aussi être une simple application console.
  • La couche de la logique métier (BLL : Business logic layer) La couche de la logique métier implémente les fonctionalités métier de l'application.
  • La couche d'accès aux données (DAL : Data access layer) : Les composants des accès aux données dans cette couche de données sont responsables d'exposer les données stockées dans ces bases de données à la couche métier.
  • La couche d'entié (ou modèle) : Cette couche passera les valeurs de la couche UI vers la BLL ou de la BLL vers la DAL selon les besoins.

Dans une architecture multi-niveaux, il n'y a pas d'interaction directe entre la couche UI et la DAL. La couche UI passera par la BLL pour effectuer une opération sur la base de données.
